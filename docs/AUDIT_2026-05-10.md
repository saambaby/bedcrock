# Bedcrock Audit ŌĆö 2026-05-10

Second-pass review against the v0.1 codebase, the `bedcrock-plan.md` spec, the existing `docs/AUDIT.md`, and a research synthesis on a parallel "Proxy Bot" build plan that surfaced 23 typed notes' worth of findings on IBKR/`ib_async`/13F/ARK/scoring/safety. Where research findings apply to bedcrock, they are folded in below.

This audit does **not** restate AUDIT.md ŌĆö read that first; this one extends it.

---

## 1. Executive summary

Bedcrock v0.1 is a coherent, paper-ready system. The architecture is materially better than the Proxy Bot v3 plan I drafted in the worktree (inbox-then-process separation, multi-worker process layout, server-side OCO only). The two designs are not mergeable ŌĆö they're different products. **BUILD_PLAN.md in the worktree should be treated as discarded** for any decision about bedcrock; the only thing worth pulling from it are specific defect findings.

> **Resolved in v0.3.0:** the vault layer was deleted entirely; reasoning migrated to Claude Code Routines + FastAPI dashboard endpoints. References below to the vault writer, vault-as-source-of-truth, Cowork prompts, or Obsidian dashboards are historical context only ŌĆö see the `v3-status` table in `docs/AUDIT.md` and Appendix C in `bedcrock-plan.md` for the v0.3.0 architecture.

Six issues in this audit are **blockers before any live consideration**. Three are **bugs in v0.1 code** (not just plan gaps): the GTC stop omission, the websocketŌåöpolling double-create race, and the missing on-startup IBKR position reconciliation. The remaining three are configuration/dependency choices that have moved since v0.1 was written (`ib_insync` is unmaintained; `yfinance` survivorship; broker-side risk-limits should be set as a backstop for the unwired daily kill switch).

The rest are quality-of-life items that should land in v0.1.1 cleanup.

## 3. Code-level findings (in priority order)

### 3.1 ­¤ö┤ BLOCKER ŌĆö `ib_insync` is unmaintained; migrate to `ib_async==2.1.0`

**Location:** `pyproject.toml` (`ib-insync>=0.9.86`); `src/broker/ibkr.py` (imports + everywhere); `src/orders/monitor.py` (event subscriptions).

**Problem:** `ib_insync` was abandoned in 2024 after Ewald de Wit's death. The maintained successor is `ib_async` (PyPI: `ib-api-reloaded` org, current 2.1.0 published 2025-12-08). API is largely drop-in: same `IB`, `Stock`, `bracketOrder`, `connectAsync`, `orderStatusEvent`, `execDetailsEvent`, `placeOrder`. Migration is mostly an import-rename plus pinning `2.1.0`.

What you lose by staying on `ib_insync`:
- No fixes for IBKR API changes (the protocol does evolve ŌĆö TWS API 10.30+ has wire-format changes that `ib_insync` won't get patched for).
- No `disconnectedEvent` improvements that `ib_async` added for cleaner reconnect handling.
- `_keep_alive()` in `monitor.py` (lines 101-107) uses `ib.sleep(1)` + `await asyncio.sleep(0.1)` in a tight loop ŌĆö this dance is needed because `ib_insync` is sync-under-async. `ib_async` is pure asyncio and you can delete that whole helper.

**Fix:**
```diff
# pyproject.toml
- "ib-insync>=0.9.86",
+ "ib_async==2.1.0",
```
```diff
# src/broker/ibkr.py
- from ib_insync import IB, Stock, Trade
+ from ib_async import IB, Stock, Trade
```
```diff
# src/orders/monitor.py
- from src.broker.ibkr import IBKRBroker
  ...
- ib.sleep(1)
- await asyncio.sleep(0.1)
+ # Pure asyncio under ib_async ŌĆö no sleep dance needed
```
Replace `await asyncio.to_thread(ib.X, ...)` calls with the native async variants where available (`reqAccountSummaryAsync`, `reqTickersAsync`, `qualifyContractsAsync` ŌĆö all return awaitables in `ib_async`).

Risk: low. The `ib_async` README explicitly maintains API parity. Test under paper for 48 hours after the swap.

### 3.2 ­¤ö┤ BLOCKER ŌĆö bracket stop/target child orders default to DAY TIF; expire at 16:00 ET

**Location:** `src/broker/ibkr.py:175-181`.

**Problem:** `ib_insync.IB.bracketOrder()` constructs three orders (parent limit + take-profit + stop-loss) but does **not** set TIF on the children. The IBKR API default for children is the same as the parent ŌĆö and bedcrock's parent is implicitly DAY (the helper's default; no override applied at line 175). A position entered at 15:59 ET has its broker-side stop **expire one minute later at 16:00 ET**. Overnight gap risk ŌåÆ no protection at the open.

This is the same bug Proxy Bot v2 had. Research surfaced documented overnight gaps on the bedcrock-style universe (COIN -25% FTX night, ROKU -26% post-earnings) where a missing stop becomes a 20-30% loss.

**Fix:**
```python
async def submit_bracket(self, spec) -> BrokerOrder:
    ...
    bracket = ib.bracketOrder(
        action=ib_action,
        quantity=quantity,
        limitPrice=entry_limit,
        takeProfitPrice=target_price,
        stopLossPrice=stop_price,
    )

    # bracket = [parent, take_profit, stop_loss] in ib_insync's helper
    parent, tp, sl = bracket

    # Parent: keep DAY (entry_zone is a daily decision, plan ┬¦6.2)
    parent.tif = "DAY"

    # Children MUST be GTC + outsideRth ŌĆö protect overnight + pre/post-market
    for child in (tp, sl):
        child.tif = "GTC"
        child.outsideRth = True

    if client_order_id:
        parent.orderRef = client_order_id

    trades = []
    for order in bracket:
        trade = await asyncio.to_thread(ib.placeOrder, contract, order)
        trades.append(trade)
    ...
```

**Reconciler audit** (in `monitor.py::_reconcile_orders` or a new `safety/reconciler.py`): on every poll, walk `ib.openOrders()` and flag any `orderType in ("STP", "STP LMT", "TRAIL", "LMT")` whose `tif != "GTC"` AND whose parent is filled. Re-issue if found.

```python
async def audit_open_order_tifs(self) -> None:
    ib = self._ib
    for trade in ib.openTrades():
        order = trade.order
        if order.orderType in ("STP", "STP LMT", "TRAIL", "LMT") and order.parentId:
            if order.tif != "GTC":
                logger.error(
                    "stop_or_tp_not_gtc",
                    order_id=order.orderId,
                    tif=order.tif,
                    parent_id=order.parentId,
                )
                await alert_human(f"Re-issue child {order.orderId}: tif={order.tif}")
```

Tests required:
- `tests/test_orders.py` ŌĆö assert `bracket[1].tif == "GTC"` and `bracket[2].tif == "GTC"` after `submit_bracket`.
- Manual paper test: open at 15:59 ET, leave, check next morning at 09:00 ET that the broker shows the stop alive.

### 3.3 ­¤ö┤ BLOCKER ŌĆö websocket + polling fallback can create duplicate Position rows

**Location:** `src/orders/monitor.py:131-196` (websocket handlers) + `337-356` (polling reconciler) + `206-265` (`_on_entry_fill`).

**Problem:** The flow is:
1. Entry fills at the broker.
2. `execDetailsEvent` fires ŌåÆ `_on_exec_details` ŌåÆ schedules `_handle_fill` task.
3. `_handle_fill` calls `_on_entry_fill` which inserts a `Position` row (line 220-235) and sets `draft.status = OrderStatus.FILLED` (line 236).

Meanwhile, the polling reconciler runs every 30s (`_poll`, line 92-99). It selects `DraftOrder.status == OrderStatus.SENT` (line 339). The polling fallback at `_reconcile_orders:337-356` checks `bo.state == BrokerOrderState.FILLED` and **calls `_on_entry_fill` again** (line 347).

Race window: between the WS handler reading `draft.status` (still SENT in its own DB session) and committing the change to FILLED at line 249, a polling pass can also see SENT, also call `_on_entry_fill`, and both will insert a `Position` row with the same `broker_order_id`.

There's no `UNIQUE` constraint on `Position.broker_order_id` (verified by grep ŌĆö would need to check `db/models.py`), and even if there is, the second insert will throw and corrupt the audit log.

**Fix #1 (idempotency at the boundary):**
```python
async def _on_entry_fill(self, db, draft, broker_order_id, ticker, filled_qty, filled_avg):
    if draft is None:
        ...

    # Idempotency check ŌĆö has a Position already been created for this broker order?
    existing = (await db.execute(
        select(Position).where(Position.broker_order_id == broker_order_id)
    )).scalar_one_or_none()
    if existing is not None:
        logger.info("entry_fill_already_processed", broker_order_id=broker_order_id)
        # Make sure draft state matches reality
        if draft.status != OrderStatus.FILLED:
            draft.status = OrderStatus.FILLED
            await db.commit()
        return

    # ... rest of the existing handler ...
```

**Fix #2 (defense in depth):** add a unique constraint on `Position.broker_order_id` so the DB rejects the second insert:

```python
# src/db/models.py ŌĆö Position class
class Position(Base):
    ...
    broker_order_id: Mapped[str | None] = mapped_column(
        String(64), unique=True, index=True
    )
```

Plus an alembic migration to add the constraint to existing data.

**Fix #3 (gate the poller):** the reconciler should also check whether a Position already exists before doing anything, not just whether the draft is SENT:

```python
async def _reconcile_orders(self, db):
    stmt = select(DraftOrder).where(DraftOrder.status == OrderStatus.SENT)
    sent = (await db.execute(stmt)).scalars().all()
    for draft in sent:
        if not draft.broker_order_id:
            continue
        # Skip if Position already created (WS handler beat us to it)
        already = (await db.execute(
            select(Position.id).where(Position.broker_order_id == draft.broker_order_id)
        )).scalar_one_or_none()
        if already:
            draft.status = OrderStatus.FILLED  # repair drift
            await db.commit()
            continue
        # ... existing logic ...
```

Tests required:
- `tests/test_orders.py::test_double_fill_idempotency` ŌĆö call `_on_entry_fill` twice with the same broker_order_id, assert exactly one Position row.
- `tests/test_orders.py::test_ws_and_poll_concurrent` ŌĆö simulate concurrent task firing, assert exactly one Position row.

### 3.4 ­¤¤Ā HIGH ŌĆö no IBKR-side position reconciliation on startup

**Location:** Missing ŌĆö should live in `LiveMonitor.start()` before the event loop begins.

**Problem:** If the worker crashes between `await asyncio.to_thread(ib.placeOrder, ...)` (broker accepts the order) and `_on_entry_fill` (DB row written), restart leaves the system in an inconsistent state: broker has the position, DB doesn't, the morning gameplan has no idea it's there. Bedcrock's whole operating model assumes the DB reflects broker truth. (v0.3.0: the morning gameplan is now produced by the `morning-analyze` Claude Code skill reading the FastAPI dashboard endpoints, not Cowork.)

**Fix:** on `LiveMonitor.start()`, after broker connect, run a reconciliation pass:

```python
async def _reconcile_against_broker(self, db: AsyncSession) -> None:
    """On startup: any IBKR position not in our DB ŌåÆ alert + create reconciliation row."""
    ib = self._broker._ib
    ibkr_positions = {p.contract.symbol: p for p in ib.positions()}
    db_positions = {
        p.ticker: p for p in
        (await db.execute(select(Position).where(Position.status == PositionStatus.OPEN)))
            .scalars().all()
    }

    for symbol, ibp in ibkr_positions.items():
        if symbol not in db_positions:
            logger.error("orphan_ibkr_position", symbol=symbol, qty=ibp.position)
            await post_position_alert(
                title=f"ŌÜĀ’ĖÅ ORPHAN: {symbol}",
                description=f"IBKR shows {ibp.position} shares; bedcrock DB has no record.",
                color=0xFBBF24,
            )
            db.add(AuditLog(
                actor="reconciler",
                action="orphan_ibkr_detected",
                target_kind="position",
                target_id=symbol,
                details={"ibkr_qty": ibp.position},
            ))

    for symbol, dbp in db_positions.items():
        if symbol not in ibkr_positions:
            logger.warning("stale_db_position", symbol=symbol)
            dbp.status = PositionStatus.CLOSED
            dbp.close_reason = CloseReason.EXTERNAL  # add to enum
            db.add(AuditLog(
                actor="reconciler",
                action="closed_externally",
                target_kind="position",
                target_id=str(dbp.id),
                details={"ticker": symbol},
            ))

    await db.commit()
```

Wire into `LiveMonitor.start()` after `await self._broker.connect()`, before `_poll` and `_keep_alive` start.

The plan ┬¦5.6 implies this exists ("Re-reads `02 Open Positions/` every 30 seconds so newly-opened positions get monitored automatically") but the current implementation only re-reads to *attach* monitors, not to *check* for orphans.

### 3.5 ­¤¤Ā HIGH ŌĆö daily kill switch is wired but never trips (AUDIT.md ┬¦scoring acknowledges)

**Location:** `src/scoring/gates.py:54` (returns unconditional `blocked=False`). The AUDIT.md ┬¦scoring says: "`daily_kill_switch` reads `daily_pnl_pct` from a context value that nobody currently populates. ... v0.1 leaves it at 0, so the gate effectively never trips."

**Mitigation in v0.1 (interim):** AUDIT.md says "relies on broker-side risk limits; user should set IBKR account-level risk limits as a backstop." That note is buried ŌĆö it should be a live runbook step, not a footnote. Confirm what's actually set in the IBKR account via TWS/Gateway ŌåÆ Account Window ŌåÆ Risk Limits.

**Fix for v0.1.1:** the EquitySnapshot table already exists. The `eod_worker` should write a daily snapshot at the start of each session, and the `monitor_worker` should compute `daily_pnl_pct = (current_equity - start_of_day_equity) / start_of_day_equity` and stash it in a process-level cache (or a small `daily_state` table) that the gate evaluator reads. Concretely:

```python
# src/workers/monitor_worker.py ŌĆö every 60s during market hours:
async def update_daily_pnl(db):
    today_start = (await db.execute(
        select(EquitySnapshot)
        .where(EquitySnapshot.mode == settings.mode)
        .where(EquitySnapshot.date == date.today())
        .order_by(EquitySnapshot.created_at.asc())
    )).scalar_one_or_none()
    if today_start is None:
        return
    account = await broker.get_account()
    pnl_pct = float((account.equity - today_start.equity) / today_start.equity * 100)
    await db.execute(
        update(DailyState)
        .where(DailyState.date == date.today(), DailyState.mode == settings.mode)
        .values(daily_pnl_pct=pnl_pct, updated_at=datetime.now(UTC))
    )
    await db.commit()

# src/scoring/gates.py ŌĆö replace the stub:
async def _gate_daily_kill_switch(self, db) -> GateResult:
    state = (await db.execute(
        select(DailyState).where(
            DailyState.mode == settings.mode,
            DailyState.date == date.today(),
        )
    )).scalar_one_or_none()
    if state is None:
        return GateResult(gate=GateName.DAILY_KILL_SWITCH, blocked=False)
    if state.daily_pnl_pct <= -settings.risk_daily_loss_pct:
        return GateResult(
            gate=GateName.DAILY_KILL_SWITCH,
            blocked=True,
            reason=f"Daily P&L {state.daily_pnl_pct:.2f}% Ōēż -{settings.risk_daily_loss_pct}%",
            overrideable=False,
        )
    return GateResult(gate=GateName.DAILY_KILL_SWITCH, blocked=False)
```

Until that lands, **set the IBKR account-side daily-loss limit manually** (TWS ŌåÆ Configure ŌåÆ Account ŌåÆ Risk Limits ŌåÆ Daily Loss = 2% of NLV). That gives you the safety net the code currently doesn't.

### 3.7 ­¤¤Ī MEDIUM ŌĆö `yfinance` for OHLCV fallback (survivorship bias)

**Location:** `pyproject.toml`; `src/ingestors/ohlcv.py` (Polygon ŌåÆ yfinance fallback per AUDIT.md ┬¦ingestors).

**Problem:** Research surfaced that yfinance overstates returns for speculative-universe backtests by 1-4%/yr because delisted symbols disappear. For bedcrock's *live* indicator computation this barely matters (you only need recent data for live tickers), but if you ever build a backtest pass against historical signals, yfinance will systematically flatter results.

**Fix (live):** acceptable as-is. Document in `src/ingestors/ohlcv.py` that this fallback is *for live indicator computation only*, not for any historical evaluation.

**Fix (future backtest):** if/when a backtest framework is added (currently absent ŌĆö see ┬¦3.10), use Norgate or CRSP instead of yfinance. This is a future-you problem, not v0.1.

### 3.10 ­¤¤Ī MEDIUM ŌĆö No backtest, no slippage model, no SPY baseline currently writing

**Location:** `bedcrock-plan.md ┬¦6.3` describes a slippage model (limit fill on cross-through, market fill at ask+5bps, +10bps for ADV<500k) and ┬¦6.5 prescribes a SPY equal-weight baseline. Neither is in the v0.1 codebase.

**Why it matters:** Phase-2 graduation criteria (┬¦9: Sharpe>1.0, Ōēź50 trades, beat SPY risk-adjusted) cannot be measured without (a) a SPY parallel portfolio writing equity snapshots and (b) realistic slippage so the paper Sharpe ŌēĀ live Sharpe by < 30%.

**Fix:** at least the SPY baseline should land before going live.
- Add a parallel `mode = baseline` portfolio in EquitySnapshot.
- On every paper entry, write a corresponding `mode=baseline` SPY shadow position with same dollar size.
- On every paper exit, close the shadow at the same date. Difference between paper return and SPY-shadow return is your alpha attribution.

Slippage model is harder ŌĆö IBKR paper has its own (optimistic, per research) simulator. Worth flagging in the post-Live Phase 1 retrospective if Sharpe degrades > 30%.

## 4. Where Proxy Bot v3 plan was wrong-for-bedcrock (don't pull these)

Some BUILD_PLAN.md ideas would actively *harm* bedcrock if back-ported:

- **Tier 2 WebSocket emergency-close monitor** ŌĆö bedcrock chose broker-side OCO only, which is *safer* (no double-sell race possible). Don't add a software stop monitor.
- **`anthropic` SDK inline call per signal** ŌĆö bedcrock's reasoning layer is decoupled from the worker process (v0.1/0.2: Cowork-via-vault; v0.3.0: Claude Code Routines hitting FastAPI). Either way: don't move Claude into the worker process.
- **SQLite** ŌĆö bedcrock is correct on Postgres for multi-worker concurrency.
- **Telegram approval** ŌĆö Discord is fine; bedcrock's signed-deep-link UI is actually better.
- **Convergence-scoring v3 with threshold 35** ŌĆö bedcrock's score range is 0-10 with cluster/insider/flow components; the convergence multiplier idea was for a 3-source model that bedcrock doesn't have.
- **8-week paper-trading minimum** ŌĆö bedcrock's 90-day Phase 2 with Ōēź50 trades is the more rigorous gate. Don't shorten it.
- **Polygon as fallback for delayed paper data** ŌĆö bedcrock doesn't subscribe per-ticker market data; the issue doesn't exist for the swing-trade cadence (Routine-driven in v0.3.0; Cowork-driven in v0.1/0.2).
